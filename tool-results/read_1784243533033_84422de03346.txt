    35→## 1. Общее описание системы
    36→
    37→**OMiK_VSM** — корпоративная информационная система для отдела мобилизации и координации персонала строительной компании «ВелесстройМонтаж» и связанных юрлиц (ВСМ, СК и др.). Система обеспечивает:
    38→
    39→- **Управление основной базой сотрудников** (112 000+ записей из выгрузки 1С)
    40→- **Ежедневный учёт персонала** (подвкладки: На площадке / Дома / Статистика)
    41→- **Кандидаты** (На сегодня, Дни оформления, Общая статистика; сопоставление с БД по ФИО+ДР, затем паспорт)
    42→- **Статистику по площадкам** (KPI: на площадке, дома, кандидаты, АУП/ИТР/РОП)
    43→- **Календарь прилёта–вылета**
    44→- **Учёт затрат на авиа-/ЖД-билеты** (организации ВСМ и СК; отдельная очередь загрузки)
    45→- **Хронологию Приёма, Перевода и Увольнения** (v4.1: `employee_chronology.py` + `hr_events_store.py`)
    46→- **Оценки сотрудников** (реестры по полугодиям, сравнение с БД)
    47→- **Работу с Excel** (загрузка, просмотр, объединение, подготовка файлов, VBA-лаборатория)
    48→- **Интеграцию с Outlook** (сканирование почты и вложений — только Windows)
    49→- **Многопользовательский режим** с RBAC (роли, группы, доступ к площадкам)
    50→- **Управление пользователями** и отдельный пароль панели «Управление данными»
    51→
    52→Система построена как **веб-приложение на главном экране (Welcome)** с карточками модулей: после логина открывается трёхколоночный дашборд, модули открываются поверх (keep-alive: панель скрывается, не размонтируется). Тяжёлая обработка Excel и доменных данных выполняется в **Python FastAPI excel-service** на порту **3031**; UI и прокси — **Next.js 16** на порту **3000**.
    53→
    54→**Платформа:** Windows 11, нативный запуск без Docker/Kubernetes/Podman. Формат дат во всём UI: **ДД.ММ.ГГГГ**.

   125→### 2.3. Ключевые архитектурные решения
   126→
   127→1. **Гибрид Next.js + Python FastAPI** — UI и сессии в Next.js; вся доменная логика Excel/SQLite — в `mini-services/excel-service/`. UI ходит только через `/api/excel/*`.
   128→
   129→2. **Единая SQLite `db/custom.db`** — Prisma и Python через `unified_db.py`. Логические имена реестра (например `employees`) переписываются в физические `{instance_id}_employees` (word-boundary).
   130→
   131→3. **Один worker uvicorn** — параллельные тяжёлые запросы встают в очередь; UI может «зависать», если с главной уходит лавина запросов.
   132→
   133→4. **Сессии в таблице `Session`** — не в JSON-файлах `.sessions/` (каталог может остаться legacy). Cookie `omik_session`, TTL порядка 8 часов.
   134→
   135→5. **Три уровня безопасности** — middleware cookie → проверка сессии/CSRF/RBAC в route handlers → `X-OMIK-Token` (= `OMIK_API_SECRET`) + role/sites в Python.
   136→
   137→6. **Keep-alive панелей** — при смене модуля панель `hidden`, а не unmount — сохраняется состояние и кэш.
   138→
   139→7. **Ленивая загрузка тяжёлых подвкладок** — в `DailyAccountingPanel`, `CandidatesPanel` грузится только активная подвкладка (цель перехода ~1 с).
   140→
   141→8. **Загрузки Excel** — `%LOCALAPPDATA%\OMiK_VSM\data\` (+ `upload/`). Очередь «Затрат по билетам» изолирована в `tickets_costs_vsm/sk.sqlite`, не из общего `upload/`.

   255→## 4. База данных (Prisma + Unified SQLite)
   256→
   257→### 4.1. Подключение
   258→
   259→| Параметр | Значение |
   260→|----------|----------|
   261→| **Провайдер** | SQLite |
   262→| **Файл** | `db/custom.db` |
   263→| **Prisma URL (Windows)** | `DATABASE_URL=file:C:/My_Project/omik-project/db/custom.db` (**не** `file:/C:/...`) |
   264→| **Python** | `OMIK_UNIFIED_DB` / `unified_db.py` |
   265→| **Данные Excel** | `%LOCALAPPDATA%\OMiK_VSM\data\` (`OMIK_DATA_DIR`) |
   266→
   267→### 4.2. Модели Prisma
   268→
   269→#### Session — сессии пользователей
   270→
   271→| Поле | Тип | Описание |
   272→|------|-----|----------|
   273→| `id` | String (@id) | 64-char hex token (значение cookie) |
   274→| `login` | String | Логин |
   275→| `role` | String | Роль |
   276→| `access` | String | Уровень доступа |
   277→| `sites` | String | Площадки через запятую |
   278→| `createdAt` | DateTime | Создание |
   279→| `expiresAt` | DateTime | Истечение (индекс) |
   280→
   281→#### ExcelFile — реестр Excel в UI
   282→
   283→| Поле | Тип | Описание |
   284→|------|-----|----------|
   285→| `id` | String (cuid) | ID |
   286→| `name` / `originalName` | String | Имена |
   287→| `path` | String | Путь |
   288→| `size` / `sheetCount` | Int | Метаданные |
   289→| `isActive` | Boolean | Активен |
   290→| `description` | String | Описание |
   291→| Связи | `macros`, `operations` | Cascade |
   292→
   293→#### Macro / MacroRun — VBA/Python макросы и запуски
   294→
   295→| Модель | Ключевые поля |
   296→|--------|----------------|
   297→| Macro | `name`, `code`, `language` (vba/python), `fileId?`, `isGlobal` |
   298→| MacroRun | `macroId`, `status`, `output`, `duration` |
   299→
   300→#### Operation — аудит операций над файлом
   301→
   302→| Поле | Описание |
   303→|------|----------|
   304→| `type` | sort, filter, merge, find_replace, pivot, … |
   305→| `params` | JSON-строка |
   306→| `status` | pending / completed / error |
   307→| `fileId` | Связь с ExcelFile |
   308→
   309→#### MainDatabase — legacy-метаданные
   310→
   311→Модель в schema остаётся для совместимости; **реальные данные сотрудников** живут в Python-слое (`main_db.py`) в физических таблицах unified DB, не как строки Prisma.
   312→
   313→### 4.3. Где лежат рабочие данные (Python)
   314→
   315→| Реестр | Модуль | UI |
   316→|--------|--------|-----|
   317→| Основная БД сотрудников | `main_db.py`, `main_db_registry.py` | Сотрудники |
   318→| Ежедневный учёт | `daily_tracking.py` | Ежедневный учёт, Welcome, Кандидаты, Площадки |
   319→| HR-события / хронология | `hr_events_store.py`, `employee_chronology.py` | Хронология |
   320→| Календарь | `calendar_db.py` | Календарь Прилет-Вылет |
   321→| Билеты ВСМ/СК | `tickets_costs.py`, `tickets_db.py` | Затраты по билетам |
   322→| Оценки | `evaluations_db.py` | Оценки сотрудников |
   323→| Пользователи | `users_store.py` | Управление данными → Пользователи |
   324→
   325→**Важно:** «Очистить очередь загрузки» ≠ «Очистить реестр» (билеты). Очередь изолирована в `tickets_costs_vsm/sk.sqlite`.

   329→## 5. Аутентификация и RBAC
   330→
   331→### 5.1. Поток входа
   332→
   333→```
   334→Пользователь → AppLoginBar (Welcome)
   335→  ↓ POST /api/excel/auth/login { login, password }
   336→  ↓ bcrypt verify (users_store)
   337→  ↓ создание Session в custom.db
   338→  ↓ Cookie: omik_session
   339→  ↓ GET /api/excel/auth/me → auth-store
   340→  ↓ рендер модулей (только при isAuthed)
   341→```
   342→
   343→Логин администратора: **`Admin`** (с заглавной **A**). При первом создании пароль пишется в `.first_admin_password`.
   344→
   345→### 5.2. Middleware и CSRF
   346→
   347→**Файл:** `src/middleware.ts`
   348→
   349→- Для `/api/*` требуется cookie сессии (исключения: login, health, публичные stubs).
   350→- Мутации проверяют CSRF через `/api/excel/auth/csrf`.
   351→
   352→### 5.3. Прокси к Python
   353→
   354→Next.js добавляет заголовки:
   355→
   356→| Заголовок | Содержимое |
   357→|-----------|------------|
   358→| `X-OMIK-Token` | `OMIK_API_SECRET` (≥ 32 символа) |
   359→| `X-OMIK-Role` | Роль из Session |
   360→| `X-OMIK-Sites` | Список площадок пользователя |
   361→
   362→Python (`auth_middleware.py`, `rbac.py`) проверяет токен и фильтрует данные по роли/площадкам.
   363→
   364→### 5.4. Роли
   365→
   366→| Роль | Доступ |
   367→|------|--------|
   368→| **Administrator / Admin** | Полный доступ, пользователи, настройки |
   369→| **ЦОК / COK / Manager** | Расширенный доступ к площадкам |
   370→| **Viewer / обычный** | Только назначенные площадки |
   371→
   372→### 5.5. Пароль «Управление данными»
   373→
   374→Отдельный пароль настроек (файл `.settings.hash` / `settings-password.ts`) — **не путать** с логином приложения. Не сбрасывается при logout UI.

   378→## 6. SQLite WAL и единая БД
   379→
   380→### 6.1. Unified DB
   381→
   382→**Файл:** `mini-services/excel-service/unified_db.py`
   383→
   384→- Prisma и Python открывают один `custom.db`.
   385→- Реестр логических имён → физические таблицы с префиксом `instance_id`.
   386→- `pandas.to_sql` — через raw `sqlite3` и **физическое** имя таблицы.
   387→
   388→### 6.2. WAL
   389→
   390→Для многопользовательского доступа рекомендуется WAL (readers не блокируют writers полностью). При экстремальной нагрузке возможны `database is locked` — excel-service работает с **одним worker**, что снижает конкурентную запись, но удлиняет очередь запросов.
   391→
   392→### 6.3. Данные вне custom.db
   393→
   394→Часть legacy/рабочих файлов остаётся в `%LOCALAPPDATA%\OMiK_VSM\data\` и постепенно мигрирует в unified-схему.

   877→## 18. Известные проблемы и ограничения
   878→
   879→### Производительность
   880→
   881→1. **Один worker uvicorn** — тяжёлые запросы сериализуются; UI может ждать минуты при лавине с Welcome.
   882→2. **Полные сканы daily/main DB** в Python вместо SQL-агрегаций.
   883→3. **Дубль welcome-dashboard** (Center + Sidebar) и несколько параллельных запросов ActiveReports.
   884→4. **Кэш только in-memory** на вкладке браузера — после F5 cold start.
   885→
   886→### Функциональные
   887→
   888→5. **Часть карточек** — заглушки (Карнет, Уволенные, Затраты на трудоустройство, Мобилизация, Utilites).
   889→6. **Outlook** — только Windows + установленный Outlook.
   890→7. **Секреты** — без `OMIK_API_SECRET` excel-service не принимает проксированные запросы.
   891→
   892→### Архитектурные отличия от «VSM Node-only» ветки
   893→
   894→В отличие от документации `VSM_DOCUMENTATION2.md` (где FastAPI декомиссирован, а Excel в `excel-server.ts`):
   895→
   896→- в **omik-project** Python excel-service **активен и обязателен**;
   897→- нет Worker Thread Pool / DuckDB / месячной архивации как в той ветке;
   898→- auth — **своя Session-таблица**, не NextAuth.js.
