    37→## 1. Общее описание системы
    38→
    39→**VSM ОМиК** — корпоративная информационная система для отдела мобилизации и координации персонала строительной компании «ВелесстройМонтаж». Система обеспечивает:
    40→
    41→- **Управление базой данных сотрудников** (113,000+ записей из 1С)
    42→- **Учёт затрат на авиа- и ЖД-билеты** (организации ВСП и СК)
    43→- **Ежедневный учёт персонала** (на объекте, в командировке, в отпуске)
    44→- **Календарь прилёта-вылета** (отслеживание перемещений)
    45→- **Автоматизированную обработку данных** (24 Python-утилиты)
    46→- **Файловый менеджер** (загрузка, просмотр, удаление, каталогизация)
    47→- **Систему оценок KPI** (рейтинг персонала)
    48→- **Генерацию отчётов** (по трудоустройству, увольнениям, мобилизации)
    49→- **Многопользовательский режим** — до 20 одновременных пользователей с изоляцией рабочих пространств
    50→- **Аутентификацию** — вход по логину/паролю (NextAuth.js + bcrypt)
    51→
    52→Система построена как **одностраничное приложение (SPA)** с боковой навигацией и динамической загрузкой панелей. Основные данные сотрудников хранятся в памяти сервера (in-memory cache) для быстрого доступа; метаданные файлов, пользователи и аудит-логи — в SQLite через Prisma ORM.

   127→### 2.3. Ключевые архитектурные решения
   128→
   129→1. **In-memory кеш вместо полной БД** — Основная база сотрудников (113K+ записей) хранится в оперативной памяти через `globalThis` для переживания HMR. SQLite используется для метаданных файлов, пользователей и аудита.
   130→
   131→2. **JSON+base64 вместо multipart/form-data** — Caddy-прокси ломает multipart-запросы, поэтому все загрузки файлов используют JSON с base64-кодированием.
   132→
   133→3. **Динамический импорт панелей** — Все панели загружаются через `next/dynamic` с `ssr: false` для уменьшения размера начального бандла.
   134→
   135→4. **Декомиссия FastAPI** — Изначально система имела Python FastAPI бэкенд на порту 3031. Вся логика перенесена в `excel-server.ts` (Node.js), Python-бэкенд не запускается.
   136→
   137→5. **Worker Thread для тяжёлых операций** — Загрузка основной базы данных вынесена в Worker Thread для предотвращения блокировки event loop.
   138→
   139→6. **SQLite WAL + retry** — Для поддержки 20 одновременных пользователей включён WAL-режим, `busy_timeout=10с`, `synchronous=NORMAL`, и реализован `withRetry()` wrapper с экспоненциальной задержкой (см. раздел 6).
   140→
   141→7. **Многопользовательская изоляция** — Каждая модель данных имеет поле `userId` для привязки к пользователю. Поле `workspace` на User позволяет группировать пользователей по рабочим пространствам (например, `vsm`, `sk`, `default`).

   549→## 5. Аутентификация и многопользовательский режим
   550→
   551→### 5.1. NextAuth.js — Credentials Provider
   552→
   553→**Файл:** `src/app/api/auth/[...nextauth]/route.ts`
   554→
   555→Система аутентификации построена на **NextAuth.js v4** с провайдером **Credentials** (логин/пароль).
   556→
   557→**Конфигурация:**
   558→
   559→```typescript
   560→export const authOptions: NextAuthOptions = {
   561→  providers: [
   562→    CredentialsProvider({
   563→      credentials: { username, password },
   564→      async authorize(credentials) {
   565→        // 1. Поиск пользователя в БД (с retry для WAL)
   566→        const user = await withRetry(() =>
   567→          db.user.findUnique({ where: { username: credentials.username } })
   568→        )
   569→        // 2. Проверка пароля через bcrypt
   570→        const isValid = await bcrypt.compare(credentials.password, user.passwordHash)
   571→        // 3. Обновление lastLoginAt
   572→        // 4. Возврат { id, name, role, workspace }
   573→      },
   574→    }),
   575→  ],
   576→  session: { strategy: 'jwt', maxAge: 8 * 60 * 60 }, // 8 часов
   577→  pages: { signIn: '/login' },
   578→  secret: process.env.NEXTAUTH_SECRET,
   579→}
   580→```
   581→
   582→**Поток аутентификации:**
   583→
   584→```
   585→Пользователь → /login (страница входа)
   586→  ↓ ввод логина/пароля
   587→  ↓ signIn('credentials', { redirect: false })
   588→  ↓ POST /api/auth/callback/credentials
   589→  ↓ authorize() → bcrypt.compare() → JWT token
   590→  ↓ Cookie: next-auth.session-token
   591→  ↓ Redirect → / (защищённая страница)
   592→```

   700→## 6. SQLite WAL: настройка для 20 пользователей
   701→
   702→### 6.1. Проблема
   703→
   704→SQLite по умолчанию использует режим `DELETE` journal, при котором:
   705→- Писатель блокирует всю БД для других писателей и читателей
   706→- При 20 одновременных пользователях возникают ошибки `SQLITE_BUSY`
   707→- Высокая вероятность «database is locked»
   708→
   709→### 6.2. Решение
   710→
   711→**Файл:** `src/lib/db.ts`
   712→
   713→Включены следующие PRAGMA при инициализации:
   714→
   715→```typescript
   716→const SQLITE_PRAGMAS = [
   717→  `PRAGMA journal_mode=WAL`,        // Readers не блокируют writers (и наоборот)
   718→  `PRAGMA busy_timeout=10000`,      // Ждать до 10с вместо немедленной ошибки SQLITE_BUSY
   719→  `PRAGMA synchronous=NORMAL`,      // Безопасно в WAL, быстрее чем FULL
   720→  `PRAGMA cache_size=-20000`,       // 20MB кеш страниц (по умолчанию 2MB)
   721→  `PRAGMA temp_store=MEMORY`,       // Временные таблицы и индексы в RAM
   722→  `PRAGMA mmap_size=268435456`,     // Memory-mapped I/O до 256MB
   723→  `PRAGMA foreign_keys=ON`,         // Включить внешние ключи
   724→]
   725→```
   726→
   727→**Инициализация** запускается один раз при первом обращении:
   728→
   729→```typescript
   730→let prismaInitialized = false
   731→
   732→async function initSqlite() {
   733→  if (prismaInitialized) return
   734→  prismaInitialized = true
   735→  for (const pragma of SQLITE_PRAGMAS) {
   736→    try {
   737→      await db.$executeRawUnsafe(pragma)
   738→    } catch {
   739→      // Игнорируем — возможно, не SQLite
   740→    }
   741→  }
   742→}
   743→
   744→// Fire and forget
   745→initSqlite().catch(() => {})
   746→```
   747→
   748→### 6.3. withRetry() — retry wrapper для операций
   749→
   750→Даже с `busy_timeout` SQLite может периодически выбрасывать «database is locked» при высокой нагрузке. `withRetry()` реализует retry с экспоненциальной задержкой:
   751→
   752→```typescript
   753→export async function withRetry<T>(
   754→  fn: () => Promise<T>,
   755→  options: {
   756→    retries?: number       // default: 5
   757→    baseDelayMs?: number   // default: 50
   758→    maxDelayMs?: number    // default: 2000
   759→  } = {},
   760→): Promise<T> {
   761→  for (let attempt = 0; attempt <= retries; attempt++) {
   762→    try {
   763→      return await fn()
   764→    } catch (error) {
   765→      const isLockError = msg.includes('locked') || msg.includes('busy')
   766→      if (!isLockError || attempt === retries) throw error
   767→
   768→      // Экспоненциальная задержка с джиттером
   769→      const delay = Math.min(
   770→        maxDelayMs,
   771→        baseDelayMs * Math.pow(2, attempt) + Math.random() * 50
   772→      )
   773→      await new Promise(r => setTimeout(r, delay))
   774→    }
   775→  }
   776→}
   777→```
   778→
   779→**Паттерны retry-ошибок:**
   780→- `locked`
   781→- `busy`
   782→- `sqlite_busy`
   783→- `database is locked`
   784→- `could not obtain lock`
   785→- `transaction conflicts`
   786→
   787→**Задержки (с базой 50ms):**
   788→| Попытка | Задержка |
   789→|---------|----------|
   790→| 1 | ~50ms + джиттер |
   791→| 2 | ~100ms + джиттер |
   792→| 3 | ~200ms + джиттер |
   793→| 4 | ~400ms + джиттер |
   794→| 5 | ~800ms + джиттер |
   795→
   796→### 6.4. withRetryTx() — retry для транзакций
   797→
   798→```typescript
   799→export async function withRetryTx<T>(
   800→  fn: (tx) => Promise<T>,
   801→  options?: { retries?: number; baseDelayMs?: number; maxDelayMs?: number },
   802→): Promise<T> {
   803→  return withRetry(() => db.$transaction(fn), options)
   804→}
   805→```
   806→
   807→### 6.5. Рекомендации по использованию
   808→
   809→**Всегда оборачивайте БД-операции в `withRetry`:**
   810→
   811→```typescript
   812→// ❌ Плохо — может упасть при 20 пользователях
   813→const user = await db.user.findUnique({ where: { id } })
   814→
   815→// ✅ Хорошо — retry при блокировке
   816→const user = await withRetry(() => db.user.findUnique({ where: { id } }))
   817→```
   818→
   819→**Для транзакций:**
   820→
   821→```typescript
   822→const result = await withRetryTx(async (tx) => {
   823→  const user = await tx.user.create({ data: { ... } })
   824→  await tx.fileRecord.create({ data: { userId: user.id, ... } })
   825→  return user
   826→})
   827→```
   828→
   829→### 6.6. Сравнение режимов SQLite
   830→
   831→| Режим | Писателей | Читателей | Изоляция | Стоимость | Подходит для |
   832→|-------|-----------|-----------|----------|-----------|-------------|
   833→| DELETE (default) | 1 (блокирует всё) | 0 во время записи | Полная | 0₽ | 1-3 пользователя |
   834→| **WAL (наш выбор)** | **1, но не блокирует чтение** | **Много** | **Snapshot** | **0₽** | **До 20 пользователей** |
   835→| PostgreSQL | Много | Много | MVCC | Платно/RU-доступ | 50+ пользователей |
   836→
   837→### 6.7. Файлы WAL
   838→
   839→При включении WAL создаются два дополнительных файла рядом с `custom.db`:
   840→
   841→| Файл | Назначение |
   842→|------|-----------|
   843→| `db/custom.db` | Основная база данных |
   844→| `db/custom.db-wal` | Write-Ahead Log (журнал операций) |
   845→| `db/custom.db-shm` | Shared memory index для WAL |
   846→
   847→**Важно:** Не удаляйте `.wal` и `.shm` файлы во время работы приложения. При корректном закрытии SQLite автоматически выполняет checkpoint и переносит данные из WAL в основной файл.

   851→## 7. Worker Thread Pool
   852→
   853→**Файл:** `src/lib/worker-pool.ts`
   854→**Worker script:** `src/lib/workers/excel-parser.worker.ts`
   855→
   856→### 7.1. Проблема
   857→
   858→При 150+ файлов/день и файлах с 30 листами парсинг Excel в основном потоке блокирует event loop → 502 Bad Gateway для всех пользователей.
   859→
   860→### 7.2. Решение: WorkerPool
   861→
   862→Пул из N worker-потоков (по умолчанию = min(CPU cores, 4)), которые парсят Excel в фоновых потоках:
   863→
   864→```
   865→API Route → workerPool.execute(task) → Worker Thread (парсинг XLSX)
   866→         ← Promise<WorkerResult>     ← Progress → UI (onProgress callback)
   867→```
   868→
   869→### 7.3. Поддерживаемые типы задач
   870→
   871→| Type | Описание | Прогресс |
   872→|------|----------|----------|
   873→| `parse-excel` | Универсальный парсинг (все листы) | По листам |
   874→| `parse-daily` | Ежедневный учёт (1 лист) | По завершении |
   875→| `parse-calendar` | Календарь (до 30 листов) | По листам |
   876→| `parse-tickets` | Билеты ВСП/СК | По завершении |
   877→
   878→### 7.4. API WorkerPool
   879→
   880→```typescript
   881→import { workerPool, type WorkerTask } from '@/lib/worker-pool'
   882→
   883→const result = await workerPool.execute({
   884→  id: 'task-1',
   885→  type: 'parse-calendar',
   886→  payload: { filePath: '/data/calendar.xlsx' },
   887→  onProgress: (percent, message) => {
   888→    console.log(`${percent}%: ${message}`) // "50%: Лист 15/30: Июль"
   889→  },
   890→  signal: abortController.signal, // Опциональная отмена
   891→})
   892→// result: { taskId, success, data, error?, duration }
   893→```
   894→
   895→### 7.5. Ключевые возможности
   896→
   897→- **FIFO очередь** — задачи ждут свободного worker'а
   898→- **Progress reporting** — callback `onProgress` из worker'а (критично для 30-листовых файлов)
   899→- **AbortSignal** — отмена задачи до или во время выполнения
   900→- **Auto-restart** — упавший worker автоматически перезапускается
   901→- **HMR-safe singleton** — пул переживает hot reload (как `db.ts`)
   902→- **TaskId correlation** — каждое сообщение от worker'а содержит `taskId`
   903→
   904→### 7.6. Мониторинг
   905→
   906→```typescript
   907→const { total, busy, queued } = workerPool.status
   908→// { total: 4, busy: 2, queued: 5 }
   909→```

   913→## 8. DuckDB: аналитические запросы
   914→
   915→**Файл:** `src/lib/duckdb.ts`
   916→**API route:** `src/app/api/analytics/route.ts`
   917→
   918→### 8.1. Проблема
   919→
   920→SQLite оптимизирован для OLTP (точечные чтения/записи). Аналитические запросы (агрегация по миллионам строк, GROUP BY, UNION ALL) выполняются медленно.
   921→
   922→### 8.2. Решение: DuckDB
   923→
   924→DuckDB — in-process OLAP база данных, запускается без сервера, читает SQLite напрямую через `sqlite_scan()`.
   925→
   926→```typescript
   927→import { duckQuery } from '@/lib/duckdb'
   928→
   929→// Аналитика по всем рейсам за год (DuckDB в 100× быстрее SQLite)
   930→const result = await duckQuery(`
   931→  SELECT site, COUNT(*) as cnt, AVG(ticketPrice) as avg_price
   932→  FROM sqlite_scan('/path/to/custom.db', 'FlightEvent')
   933→  WHERE flightDate >= '2025-01-01' AND flightDate < '2026-01-01'
   934→  GROUP BY site
   935→  ORDER BY cnt DESC
   936→`)
   937→```
   938→
   939→### 8.3. Ключевые функции
   940→
   941→| Функция | Описание |
   942→|---------|----------|
   943→| `duckQuery(sql, params?)` | Выполнить SQL, вернуть массив строк |
   944→| `duckQueryOne(sql, params?)` | Выполнить SQL, вернуть одну строку |
   945→| `buildCrossPeriodQuery(table, opts)` | Построить UNION ALL по основной + архивным БД |
   946→| `listArchives()` | Список доступных архивов (YYYY-MM) |
   947→| `getMainDbPath()` | Путь к основной БД |
   948→| `getArchiveDbPath(ym)` | Путь к архиву |
   949→| `archiveExists(ym)` | Проверить существование архива |
   950→| `closeDuckDB()` | Закрыть соединение (graceful shutdown) |
   951→
   952→### 8.4. Cross-period запросы
   953→
   954→```typescript
   955→// Запрос по текущей БД + всем архивам за указанный период
   956→const sql = buildCrossPeriodQuery('FlightEvent', {
   957→  dateColumn: 'flightDate',
   958→  startDate: '2024-06-01',
   959→  endDate: '2025-06-30',
   960→  select: 'site, COUNT(*) as total_flights',
   961→  groupBy: 'site',
   962→})
   963→const result = await duckQuery(sql)
   964→```
   965→
   966→### 8.5. API: `/api/analytics`
   967→
   968→| Метод | Параметры | Описание |
   969→|-------|-----------|----------|
   970→| `POST` | `{ query: "SELECT ..." }` | Raw SQL (только SELECT) |
   971→| `POST` | `{ table, dateColumn, startDate, endDate, ... }` | Builder mode |
   972→| `GET` | — | Список архивов + путь к основной БД |
   973→
   974→**Безопасность:** Разрешены только `SELECT` и `WITH` запросы. INSERT/UPDATE/DELETE → 403.

   978→## 9. Архивация: месячные архивы
   979→
   980→**Файл:** `src/lib/archive.ts`
   981→**Script:** `scripts/archive.js` (standalone, better-sqlite3)
   982→**API route:** `src/app/api/archive/route.ts`
   983→
   984→### 9.1. Проблема
   985→
   986→При 150 файлов/день основная БД вырастет до миллионов строк за год → замедление запросов, раздувание файла, долгий VACUUM.
   987→
   988→### 9.2. Решение: месячные архивы
   989→
   990→Каждый 1-й день месяца данные за прошлый месяц переносятся в `archive-YYYY-MM.db`:
   991→
   992→```
   993→custom.db (текущие данные ~800K строк)
   994→├── archive-2025-01.db (read-only)
   995→├── archive-2025-02.db (read-only)
   996→├── archive-2025-03.db (read-only)
   997→└── ...
   998→```
   999→
  1000→### 9.3. Архивируемые таблицы
  1001→
  1002→| Таблица | Столбец даты | Объём/мес |
  1003→|---------|-------------|-----------|
  1004→| `DailyTrackingRecord` | `eventDate` | ~13K-52K строк (26 файлов) |
  1005→| `FlightEvent` | `flightDate` | ~10K-40K строк |
  1006→| `TicketRecord` | `flightDate` | ~5K-20K строк |
  1007→| `RatingRecord` | `createdAt` | ~1K-5K строк |
  1008→
  1009→### 9.4. Процесс архивации
  1010→
  1011→1. Создаётся `archive-YYYY-MM.db` с аналогичной схемой
  1012→2. Данные за указанный месяц COPY в архив (transaction)
  1013→3. Удаление скопированных данных из основной БД
  1014→4. `VACUUM` обеих БД для сжатия
  1015→5. Создание индексов в архиве
  1016→6. Если ошибка — partial archive удаляется (atomic)
  1017→
  1018→### 9.5. API: `/api/archive`
  1019→
  1020→| Метод | Параметры | Описание |
  1021→|-------|-----------|----------|
  1022→| `POST` | `{ yearMonth: "2025-01" }` | Архивировать конкретный месяц |
  1023→| `POST` | `{ auto: true }` | Авто-архивация прошлого месяца (для cron) |
  1024→| `GET` | — | Список архивов + размер |
  1025→| `GET` | `?yearMonth=2025-01&table=FlightEvent` | Превью записей из архива |
  1026→
  1027→### 9.6. Автоматическая архивация (Cron)
  1028→
  1029→**Расписание:** 1-е число каждого месяца в 02:00 (Europe/Moscow)
  1030→
  1031→Cron-задача вызывает `POST /api/archive {"auto":true}`.
  1032→
  1033→### 9.7. Запросы к архивам
  1034→
  1035→Архивы — read-only SQLite файлы. Запрашиваются через:
  1036→1. **DuckDB `sqlite_scan()`** — для аналитических запросов

  1038→3. **`buildCrossPeriodQuery()`** — автоматический UNION ALL по текущей + архивным БД

  1040→### 9.8. Прогноз по объёмам

  1121→

  1123→

  1125→- **Под-шаги:** Каждая задача может иметь массив `TaskStep[]` с отдельными статусами

  1853→## 21. Известные проблемы и ограничения
  1854→
  1855→### Критические
  1856→1. **502 Bad Gateway при загрузке** — Долгая обработка XLSX.read() блокирует event loop
  1857→2. **Caddy ломает multipart/form-data** — Обход через JSON+base64 (на 33% больше размер)
  1858→
  1859→### Производительность
  1860→3. **113K+ записей в памяти** — ~100+ МБ RAM
  1861→4. **Нет виртуализации таблиц** — Пагинация без виртуализации
  1862→5. **SheetJS segfault** — На больших файлах в Turbopack
  1863→
  1864→### Многопользовательский режим
  1865→6. **In-memory кеш общей базы — глобальный** — Основная база сотрудников (mainDbCache) хранится в `globalThis` и **не изолирована** между пользователями. Все пользователи видят одну и ту же загруженную базу. Изоляция `userId` применяется только к метаданным в SQLite (ExcelFile, FileRecord и т.д.).
  1866→7. **WAL checkpoint** — При большом количестве записей WAL-файл может расти. SQLite автоматически выполняет checkpoint, но при экстремальной нагрузке может потребоваться ручной `PRAGMA wal_checkpoint(TRUNCATE)`.
  1867→
  1868→### Функциональные
  1869→8. **Mock-данные** — DailyTracking, Ratings, Dashboard stats
  1870→9. **10 из 19 вкладок в разработке**
  1871→10. **Нет кнопки «Очистить»** — Для загрузок DailyTracking
  1872→11. **Нет persisted state** — Zustand stores не персистятся
